# AGENT.md

## プロジェクト名

ThinQuest

## プロジェクト概要

ThinQuest は、対話を通じて思考を鍛える RPG です。

このアプリは、ユーザーに答えを与える通常の AI チャットアプリではありません。
ユーザーがテーマやアイデアを入力し、敵キャラクターからの質問、反論、追加の突っ込みに答えながら戦う構成です。

目的は、ユーザーが自分の考えを整理し、説明力を高め、論理を強くし、批判への応答を練習できるようにすることです。

## コアコンセプト

AI は答えを与えません。
AI は、ユーザーの思考に挑戦する敵として振る舞います。

最も重要な体験は次のとおりです。

- ユーザーが自分で考える
- 敵が弱点を指摘する
- ユーザーが自分の答えを改善する
- ユーザーが EXP を得て成長する

このアプリを、単純なチャットボットやアドバイスアプリにしてはいけません。

## MVP の範囲

シンプルな 2D Unity プロトタイプを作ること。

必要な画面:

1. タイトル画面
2. ホーム画面
3. テーマ入力画面
4. 敵選択画面
5. バトル画面
6. 結果画面

必要な機能:

- テーマを入力できる
- 敵を選択できる
- 敵の反論や質問を表示する
- ユーザーが回答を入力できる
- 簡単な評価を表示する
- EXP を獲得する
- レベルアップする
- 最小限のユーザーデータをローカル保存する

## 技術方針

Unity 2022.3 LTS を使用すること。

2D Built-in Render Pipeline を使用すること。

C# を使用すること。

MVP では、ローカル LLM やオンライン AI を実装しないこと。
固定テキストまたはルールベースのテンプレートを使うこと。

AI 連携は将来のフェーズで扱うこと。

## ビジュアル・雰囲気

ThinQuest は「クエスト」体験なので、中世ファンタジー風の雰囲気を基本にすること。

ユーザーに見える UI 文言は日本語表記にすること。

画面や文言は、ギルド、酒場、城下町、羊皮紙、石壁、松明、紋章、訓練場、決闘、試練、巻物などを連想できる方向に寄せる。

ただし、派手な魔法や重厚すぎるダークファンタジーではなく、思考トレーニング用の軽い2Dプロトタイプとして読みやすさを優先する。

UI の基本トーン:

- 背景は暗い石壁や夜のギルドホールを思わせる落ち着いた色
- パネルは羊皮紙や古い紙を思わせる明るい茶色
- 強調色は金、銅、深い赤、革の茶色
- ボタン文言は Start / Home などの汎用語ではなく、ギルドへ入る、クエスト開始、相手を選ぶ、闘技場へ、などクエストらしい日本語にする
- 敵は単なるUI項目ではなく、挑戦者・門番・商人・騎士などの役割が伝わる表現にする

## レイアウト方針

画面アスペクト比や解像度が変わっても、主要コンポーネントの相対位置関係を維持すること。

背景、キャラクター、パネル、ボタン、説明文、結果表示などは、固定ピクセル座標だけに依存させず、画面幅・高さ・安全な余白・基準アンカーから位置とサイズを計算すること。

特に次の点を守ること。

- キャラクターと UI パネルが、横長・縦長・小さい画面で重ならないようにする
- 左右どちらかにキャラクターを置く画面では、反対側の UI 領域を維持する
- ボタンや重要テキストは画面外へ出さない
- 画面が狭い場合は、要素を潰すのではなく、縦積み・スクロール・省スペース表示へ切り替える
- 背景画像はアスペクト比を維持して表示し、重要な被写体が UI に隠れない配置を前提にする
- 新しい UI を追加したら、少なくとも標準横長、狭い横長、縦長に近い画面で破綻しにくいか確認する

## 敵キャラクター

### Beginner Slime

役割:
説明が分かりやすいかを確認する。

質問例:

- このアイデアは何についてのものですか？
- これは誰のためのものですか？
- もっと簡単に説明できますか？
- 具体例を 1 つ挙げられますか？

### Logic Knight

役割:
論理の弱さ、矛盾、理由不足を見つける。

質問例:

- その主張の根拠は何ですか？
- 証拠はありますか？
- 矛盾している点はありませんか？
- 逆の場合はどうなりますか？

### Realist Merchant

役割:
実用性、需要、収益性、継続性を確認する。

質問例:

- 実際にこれを使うのは誰ですか？
- なぜ人は使い続けるのですか？
- どうやって収益につながりますか？
- 本当に実現可能ですか？

### Harsh Reviewer

役割:
ユーザーの不満や使いにくさの問題を指摘する。

質問例:

- ユーザーが面倒だと感じる部分はどこですか？
- なぜユーザーは途中でやめそうですか？
- 既存の方法より本当に簡単ですか？
- 最大の弱点は何ですか？

### Ethics Guardian

役割:
安全性、誤解の可能性、倫理的な問題を確認する。

質問例:

- 誤解を招く可能性はありませんか？
- 誰かを傷つけるおそれはありませんか？
- 危険な表現は含まれていませんか？
- 安全に説明するにはどうすればよいですか？

### Cold Investor

役割:
成長性、収益性、差別化、勝ち筋を確認する。

質問例:

- 何が他と違うのですか？
- なぜ成長すると言えますか？
- 最も強い売りは何ですか？
- 競合に勝てる理由は何ですか？

## ユーザーステータス

ユーザーは 6 つの能力値を持ちます。

- 要約力
- 論理力
- 具体化力
- 反論耐性
- 説明力
- 視点切替力

各バトルでは、これらの能力値の 1 つ以上に EXP を与えること。

## 口調ルール

敵キャラクターはユーザーに厳しく問いかけてもよいが、侮辱してはいけない。

強すぎる否定表現は避けること。

悪い例:

- このアイデアは浅いです。
- これは不可能です。
- これは価値がありません。

良い例:

- 弱い点は、継続して使われる理由がまだ不明確なことです。
- この部分を説明できると、アイデアの説得力が増します。
- 具体例があると、この回答はもっと強くなります。

## 医療表現ルール

医療的な効能表現は使わないこと。

避ける表現:

- 脳機能を改善する
- 発達障害を治療する
- 失語症を改善する
- 認知機能を回復する
- 頭が良くなる

使ってよい表現:

- 思考トレーニング
- 言語練習
- 説明力トレーニング
- 議論の練習
- 考える習慣づくり
- 思考整理

## 開発優先順位

優先度 1:
遊べるプロトタイプを作る。

優先度 2:
バトルの流れを楽しくする。

優先度 3:
EXP とレベルアップのフィードバックを入れる。

優先度 4:
敵の個性を強くする。

優先度 5:
ルールベースの反論生成を追加する。

基本のバトル体験が成立する前に、システムを作り込みすぎないこと。
変更はレビューしやすい小さな単位で進めること。
一度に大きく実装するのではなく、PR単位の小さな段階実装を優先すること。

## 最初の実装方針

最初は次の Unity シーンから始めること。

- TitleScene
- HomeScene
- ThemeInputScene
- EnemySelectScene
- BattleScene
- ResultScene

まずはシンプルな UI を使うこと。

仮のプレースホルダー画像を使ってよい。

## 保存するデータ

ローカルセーブデータを使うこと。

保存対象:

- ユーザーレベル
- 累計 EXP
- 各能力値の EXP
- 各能力値のレベル
- バトル履歴
- 最後に選択した敵
- 最後に入力したテーマ

## コーディングガイドライン

シンプルで読みやすい C# を書くこと。

分かりやすいクラス名を優先すること。

不要なアーキテクチャは避けること。

まずは小さなスクリプトで構成すること。
各実装ステップは、ユーザーが確認しやすいように小さく保つこと。

想定するスクリプト名の例:

- GameManager
- UserDataManager
- SceneLoader
- EnemyData
- BattleManager
- ExpManager
- ResultManager

## 重要ルール

常に次のコアアイデアを守ること。

ThinQuest は、敵の反論に応答することで、ユーザー自身の考えを鍛えていく思考バトル RPG である。

このアプリを、受け身で答えを生成する AI ツールにしてはいけない。

GitHub への push は、ユーザーが明示的に「pushして」「プッシュして」などと依頼した場合にだけ行うこと。

修正・コミット後に追加の push が必要そうに見えても、ユーザーの明示依頼なしに勝手に push してはいけない。
